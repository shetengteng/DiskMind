### 2.3 进程模型

DiskMind 是**单应用进程**架构,但内部使用 Rust 的 Tokio 多任务调度:

| 角色 | 实现 | 数量 | 职责 |
| --- | --- | --- | --- |
| UI Process | WebView2 / WKWebView | 1 | 渲染 Vue 应用 |
| Main Process | Rust (Tauri 主进程) | 1 | IPC 路由、状态管理 |
| Scan Worker Pool | Tokio task / rayon | N (CPU 核数) | 并发遍历文件 |
| AI Worker | Tokio task | 1–2 | LLM 异步请求队列 |
| Cleaner Worker | Tokio task | 1 | 串行执行删除(保证一致性) |
| Watcher | notify crate 后台线程 | 1 (可选) | 实时变更监控 |

**关键决策**:扫描与清理**不放在 UI 线程**,通过 channel 异步通信,避免 UI 卡顿。

### 2.4 跨平台抽象层

| 抽象 | macOS 实现 | Windows 实现 |
| --- | --- | --- |
| 用户目录定位 | `~/Library/`, `~/Documents/` | `%APPDATA%`, `%LOCALAPPDATA%` |
| 应用产物路径 | Bundle 同名扫描 (Caches, Logs, Application Support) | 注册表 + Program Files + AppData 扫描 |
| 系统保护目录 | SIP 保护清单 | Windows 保护文件清单 |
| 删除操作 | `NSFileManager.trashItem` | `SHFileOperation FO_DELETE + FOF_ALLOWUNDO` |
| 通知 | `UNUserNotificationCenter` | `Toast Notifications` |
| 自动更新 | Sparkle 协议 | Tauri Updater + Windows 签名 |
| 文件监控 | FSEvents | ReadDirectoryChangesW |

**实现策略**:`trait FsAdapter { fn trash(&self, path: &Path) -> Result<()>; ... }`,在 `cfg(target_os)` 下提供不同实现。

## 3. 模块详细设计

### 3.1 Scanner — 扫描引擎

#### 3.1.1 职责
- 遍历指定目录,产出文件元数据流
- 计算文件指纹(分级哈希)
- 写入本地元数据索引
- 支持取消、暂停、增量

#### 3.1.2 调度策略

**根据存储介质区分**(关键设计):
- **SSD**:CPU 并行度 = `num_cpus`,最大化吞吐
- **HDD**:CPU 并行度 = 1–2,避免随机读放大延迟

```rust
// 伪代码
let strategy = match probe_disk_type(&root) {
    DiskType::Ssd => ScanStrategy { parallelism: num_cpus(), chunk_size: 4096 },
    DiskType::Hdd => ScanStrategy { parallelism: 2, chunk_size: 1024 },
};
```

#### 3.1.3 遍历策略
- BFS 优先,内存中维护队列上限(防爆栈)
- 跳过清单(硬编码 + 用户白名单):
  - macOS: `/System`, `/private`, `/Volumes`(外接盘需用户授权)
  - Windows: `C:\Windows\System32`, `C:\$Recycle.Bin`, `pagefile.sys`
- 软链接处理:**默认不跟随**,避免循环

#### 3.1.4 哈希策略(分级)

```
文件 ≤ 4 KB    → 全量 BLAKE3(快)
4 KB <  ≤ 64 MB → 头部 4KB SHA-1 + 尾部 4KB SHA-1 → 命中再算 BLAKE3
> 64 MB         → 头/中/尾各 4KB SHA-1 + size  → 命中再 BLAKE3
```

理由:绝大多数重复文件大小不同,二阶哈希可避免不必要的全量计算。

#### 3.1.5 增量扫描

- 上次扫描的 `mtime` 与 `size` 作为 cache key
- 若 `(mtime, size)` 未变 → 复用上次的指纹与分类结果
- 若变化 → 重新计算

#### 3.1.6 取消与进度

- `CancellationToken` 在每次循环检查
- `Progress` 通过 mpsc::Sender 实时上报到 UI(节流到 200ms 一次)

#### 3.1.7 输出数据结构

```rust
pub struct FileEntry {
    pub path: PathBuf,
    pub size: u64,
    pub mtime: i64,
    pub atime: i64,
    pub fingerprint: Option<String>,   // BLAKE3 hex
    pub mime_guess: Option<String>,
    pub owner_app: Option<String>,     // 通过路径推断的归属应用
    pub category: Option<Category>,    // 由 Classifier 填充
}
```

### 3.2 Classifier — 分类器

#### 3.2.1 分类标签集

```
- SystemJunk        (系统垃圾)
- AppCache          (应用缓存)
- AppLog            (应用日志)
- TempFile          (临时文件)
- LargeFile         (大文件 > 100MB)
- DuplicateFile     (重复文件)
- DownloadedInstaller (旧的安装包)
- BrowserCache      (浏览器缓存)
- DeveloperJunk     (Xcode/IDE 缓存)
- MediaCache        (Adobe/DaVinci 等媒体缓存)
- Unknown           (未知,送 AIEngine 判定)
- Protected         (受保护,永不建议删除)
```

#### 3.2.2 规则库结构(可热更新)

`rule-pack.json`:
```json
{
  "version": "2026.05.001",
  "rules": [
    {
      "id": "macos-photoshop-cache",
      "category": "MediaCache",
      "platform": ["macos"],
      "match": {
        "path_glob": ["**/Library/Caches/com.adobe.Photoshop/**"],
        "owner_app": "Adobe Photoshop"
      },
      "default_risk": 15,
      "user_facing_name": "Photoshop 缓存"
    }
  ]
}
```

**热更新机制**:启动时从 CDN 拉取版本号,有更新则下载新 Pack 并热加载(用户可关闭)。

#### 3.2.3 应用归属推断

通过路径模式匹配,识别文件所属应用:
- macOS: `Library/Caches/<bundle_id>/...` → 反查 bundle_id → 应用名
- Windows: 注册表 `Uninstall` 键 + 路径反向匹配

#### 3.2.4 未知文件处理
未命中任何规则 → 打标 `Unknown` → 进入 AIEngine 队列。

### 3.3 AIEngine — AI 引擎

#### 3.3.1 职责
- 接受 `Unknown` 与 `Ambiguous` 文件,产出结构化判定
- 提供对话式问答接口
- 缓存判定结果

#### 3.3.2 Provider 抽象(LiteLLM 风格)

> **设计目标**:一套统一接口,接入任意 LLM 供应商 · 用户只需配置 `base_url + api_key` 即可换源。

##### 3.3.2.1 核心 Trait

```rust
#[async_trait]
pub trait LlmProvider: Send + Sync {
    /// Provider 元信息(name / type / status)
    fn meta(&self) -> ProviderMeta;

    /// 批量分类(同步等待完整 JSON)
    async fn classify(&self, files: &[FileMeta]) -> Result<Vec<Verdict>>;

    /// 对话流(SSE)
    async fn chat_stream(
        &self,
        messages: &[Message],
        ctx: ChatContext,
    ) -> Result<BoxStream<'static, Result<ChunkEvent>>>;

    /// 连通性测试
    async fn ping(&self) -> Result<PingResult>;
}
```

##### 3.3.2.2 三类内置实现

| Provider | 实现策略 | 覆盖范围 |
| --- | --- | --- |
| `OpenAICompatibleProvider` | 用户配置 `base_url + api_key + model` | **90% 供应商**(DeepSeek / SiliconFlow / Moonshot / Together / Groq / Ollama / OpenAI / 任意自托管) |
| `AnthropicProvider` | 适配 Anthropic Messages API | Claude 3.x 系列 |
| `GeminiProvider` | 适配 Google Generative API | Gemini 1.5 / 2.x |

> Rust 实现底座:`async-openai` crate(支持自定义 base_url + headers · 已覆盖 OpenAI 兼容协议的全部能力)

##### 3.3.2.3 Provider 注册中心

```rust
pub struct ProviderRegistry {
    providers: HashMap<ProviderId, Arc<dyn LlmProvider>>,
    default: ProviderId,
    fallback_chain: Vec<ProviderId>,   // 失败自动降级序列
    router: Router,                    // 规则路由(下文)
}

impl ProviderRegistry {
    pub fn select_for(&self, req: &ClassifyRequest) -> Arc<dyn LlmProvider> {
        // 1. 检查路由规则(文件数 / 网络 / 成本)
        // 2. 未命中 → 返回 default
        if let Some(id) = self.router.route(req) {
            return self.providers[&id].clone();
        }
        self.providers[&self.default].clone()
    }
}
```

##### 3.3.2.4 路由规则(LiteLLM 思想落地)

```rust
pub enum RoutingRule {
    /// 文件数超阈值 → 切到更便宜的 Provider
    BatchSizeGT { threshold: usize, provider: ProviderId },
    /// 网络异常 → 切到本地
    NetworkOffline { provider: ProviderId },
    /// 今日成本超阈值 → 切到本地
    DailyCostExceed { cny_threshold: f64, provider: ProviderId },
    /// 自定义(后续可加)
    Custom { predicate: Box<dyn Fn(&ClassifyRequest) -> bool + Send + Sync>,
             provider: ProviderId },
}

pub struct Router {
    rules: Vec<RoutingRule>,
}
```

##### 3.3.2.5 配置示例

参见 §4.4 `config.toml` 多 providers 配置。

##### 3.3.2.6 三种工作模式(用户视角)

| 模式 | 含义 | 实现 |
| --- | --- | --- |
| Cloud | 全部走云端 | 默认 Provider = 云端(如 DeepSeek) |
| Local | 全部走本地 | 默认 Provider = Ollama |
| Hybrid(默认) | 元数据本地预筛 + 复杂决策上云 | 路由规则 + Fallback Chain |

> 这三种模式本质上是**路由规则的预设**,不再是写死的 enum。

#### 3.3.3 批量打包策略

为降低 token 成本,单次请求评估 **50 个文件**:

```
请求体 ≈ system_prompt(固定 800 tokens) + 50 × 80 tokens = ~4800 tokens
响应体 ≈ 50 × 120 tokens = ~6000 tokens
单次成本 ≈ ¥0.03(DeepSeek-V3)
```

#### 3.3.4 Prompt 模板(System)

```
你是 DiskMind 的文件分析助手。基于以下文件元数据,判断每个文件的类别、
删除风险与用户面向解释。严格输出 JSON,符合 schema。

【输入】文件数组:[{path, size_mb, mtime, owner_app_guess, mime}]

【输出 Schema】:
[{
  "path": string,
  "category": "AppCache" | "AppLog" | "TempFile" | "MediaCache"
            | "DeveloperJunk" | "DownloadedInstaller" | "Unknown",
  "risk_score": 0-100,
  "risk_level": "low" | "medium" | "high",
  "explanation": string (≤ 60 字中文),
  "action": "safe_to_delete" | "review" | "keep" | "unknown"
}]

【规则】
1. 证据不足返回 category="Unknown" + action="unknown",不要臆造
2. 系统目录、用户文档目录 → risk_score >= 80
3. 解释要让普通用户看得懂,避免技术黑话
4. 严禁建议删除任何与近 7 天访问相关的文件
```

#### 3.3.5 JSON Schema 校验

使用 `schemars` + `jsonschema` crate 做硬约束。校验失败 → 重试一次 → 仍失败 → 降级为规则模式默认值,不向用户暴露 LLM 错误。

#### 3.3.6 结果缓存

```
cache_key = sha1(fingerprint + model_version + prompt_hash)
缓存表 ai_cache(key, result_json, created_at)
TTL = 90 天
```

#### 3.3.7 对话式问答

- 独立 endpoint,使用上下文窗口长的模型
- 注入当前文件的元数据上下文
- 流式返回 SSE,UI 逐字呈现

#### 3.3.8 Fallback Chain(失败降级)

```
ProviderRegistry.classify_with_fallback(req):
  for provider in [default, ...fallback_chain]:
      try:
          result = provider.classify(req).timeout(60s).retries(3)
          return result
      catch (TransientError):
          continue   // 自动切换下一个 Provider
      catch (PermanentError):
          break      // 配置/认证错误,不重试

  fallback_to_rule_only_mode()
  ui.toast("AI 暂不可用 · 已切换为规则模式")
```

降级触发条件:
- 网络错误 / 超时
- HTTP 5xx
- HTTP 401/403 → 标记为永久错误,跳过该 Provider 不再尝试
- 速率限制 429 → 指数退避后重试

每次降级在 `ai_cache` 中记录(source = `fallback_<n>`),用于 Reports/AI 使用统计页展示"失败重试 3 次 (98% 成功)"。

### 3.4 Policy — 决策引擎

#### 3.4.1 三档策略

| 策略 | 默认勾选 | 适用 |
| --- | --- | --- |
| **Safe(安全)** | `risk_score ≤ 20` | 默认推荐给新手 |
| **Balanced(谨慎)** | `risk_score ≤ 50` | 推荐给中级用户 |
| **Aggressive(激进)** | `risk_score ≤ 80` 但需逐项确认 | 极客使用 |

#### 3.4.2 白名单(永不删除)

- 系统受保护目录
- 用户近 7 天打开过的文件(从 `Spotlight` / `Windows Search` 索引获取)
- 用户手动 pin 的文件
- 标记为"重要"的文件(macOS Finder Tag / Windows 高优先级)
- 文件大小 < 10 KB 但路径包含 `key`/`license`/`token` 等关键字

#### 3.4.3 用户偏好学习(P2 阶段)

- 记录用户每次"建议删/用户操作"的差异
- 训练简易加权模型(LightGBM 或线性回归)
- 调整 risk_score 偏移量,实现"越用越懂"

### 3.5 Cleaner — 清理执行器

#### 3.5.1 沙箱机制

**核心承诺:DiskMind 自己不直接调用 `unlink`/`DeleteFile`,而是先移入应用内 Trash**

```
用户点击"清理" 
   → Cleaner 在 ~/.diskmind/trash/<bundle_id>/ 下创建硬链接(或移动)
   → 在 SQLite 中记录 trash_manifest:
        { original_path, trash_path, deleted_at, scan_bundle_id }
   → 显示"已清理 X.XX GB,可在 Trash 中恢复"
   → 30 天后,后台 worker 异步真删
```

**Windows / macOS 区别**:
- macOS:同卷可直接 `rename`,跨卷复制+删除
- Windows:使用 `MoveFile`,跨卷使用 `MoveFileEx` 的 `MOVEFILE_COPY_ALLOWED`

#### 3.5.2 回滚

- UI 提供"Trash"视图,列出 30 天内所有"已清理"项
- 单项恢复:`rename` 回原路径,若原路径已被占用 → 提示用户重命名
- 批量恢复:按 scan_bundle_id 一次回滚整个扫描会话

#### 3.5.3 真删任务

```
每天凌晨 (或下次启动) 触发:
  - 扫描 trash_manifest 中 deleted_at > 30 天前的记录
  - 串行 unlink,记录每条结果
  - 真删失败(文件被占用)→ 延期 7 天再尝试
```

#### 3.5.4 异常处理

| 异常 | 行为 |
| --- | --- |
| 文件被占用 | 标记 `pending`,下次重试 |
| 权限不足 | UI 提示需提权,引导用户授权 |
| 路径不存在 | 视为已清理,从 manifest 移除 |
| 跨卷失败 | 改用复制+删除策略 |

### 4.2 SQLite DDL(核心表)

```sql
-- 扫描会话
CREATE TABLE scans (
  id            TEXT PRIMARY KEY,
  started_at    INTEGER NOT NULL,
  finished_at   INTEGER,
  root_paths    TEXT NOT NULL,        -- JSON
  status        TEXT NOT NULL,        -- running/completed/cancelled/failed
  total_files   INTEGER,
  scanned_bytes INTEGER
);

-- 文件元数据(主表,跨 scan 共享)
CREATE TABLE files (
  fingerprint   TEXT PRIMARY KEY,
  path          TEXT UNIQUE NOT NULL,
  size          INTEGER NOT NULL,
  mtime         INTEGER NOT NULL,
  atime         INTEGER,
  mime_guess    TEXT,
  owner_app     TEXT
);
CREATE INDEX idx_files_path ON files(path);
CREATE INDEX idx_files_size ON files(size);

-- 扫描中的文件实例(关联 scan)
CREATE TABLE file_entries (
  scan_id       TEXT NOT NULL,
  fingerprint   TEXT NOT NULL,
  category      TEXT NOT NULL,
  risk_score    INTEGER,
  risk_level    TEXT,
  explanation   TEXT,
  action        TEXT,
  source        TEXT NOT NULL,        -- rule/ai/cache
  PRIMARY KEY (scan_id, fingerprint),
  FOREIGN KEY (scan_id) REFERENCES scans(id),
  FOREIGN KEY (fingerprint) REFERENCES files(fingerprint)
);

-- AI 结果缓存
CREATE TABLE ai_cache (
  cache_key     TEXT PRIMARY KEY,     -- sha1(fingerprint + model + prompt_hash)
  result_json   TEXT NOT NULL,
  model         TEXT NOT NULL,
  created_at    INTEGER NOT NULL
);
CREATE INDEX idx_ai_cache_created ON ai_cache(created_at);

-- Trash 沙箱
CREATE TABLE trash_items (
  id            TEXT PRIMARY KEY,
  scan_id       TEXT,
  original_path TEXT NOT NULL,
  trash_path    TEXT NOT NULL,
  size          INTEGER NOT NULL,
  deleted_at    INTEGER NOT NULL,
  hard_delete_after INTEGER NOT NULL,
  status        TEXT NOT NULL        -- in_trash / hard_deleted / restored / pending
);

-- 用户白名单
CREATE TABLE whitelist (
  path_pattern  TEXT PRIMARY KEY,
  reason        TEXT,
  created_at    INTEGER NOT NULL
);

-- 用户行为(用于偏好学习)
CREATE TABLE user_actions (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  fingerprint   TEXT NOT NULL,
  suggested     TEXT NOT NULL,        -- safe_to_delete / keep / review
  user_chose    TEXT NOT NULL,        -- delete / keep / restore
  created_at    INTEGER NOT NULL
);
```

### 4.3 SQLite 分析查询(原 DuckDB 分析视图,2026-05-26 合并到 SQLite)

不再为分析查询单独引入 DuckDB,以下聚合查询直接在 SQLite 上跑(数据量 <50 万行,典型亚秒返回;若需要可按对应列加 INDEX):

```sql
-- 跨 scan 趋势(按日聚合 scan_run)
SELECT
    date(datetime(started_at / 1000, 'unixepoch')) AS day,
    SUM(total_bytes) / 1024 / 1024 / 1024 AS scanned_gb,
    COUNT(*) AS scan_count
FROM scan_run
WHERE cancelled = 0
GROUP BY day
ORDER BY day;

-- Top 类目占用(替代原 Top 占用应用,按 classifier 分类聚合)
SELECT
    category,
    SUM(size_bytes) AS total_bytes,
    COUNT(*) AS file_count
FROM scan_result
GROUP BY category
ORDER BY total_bytes DESC
LIMIT 50;
```

> 实现位置:`app/src-tauri/src/db/mod.rs` 的 `list_runs()` 即按 `run_id` 聚合 `scan_result` 计算 `categoryBreakdown` 和 `reclaimable_bytes`;前端 `stores/reports.ts` 进一步按 `finishedAt` 维度聚合得到 `trendByDay` 和 `aggregatedCategoryBreakdown`。

## 6. 关键算法

### 6.1 分级哈希(去重)

```
输入: 文件流
输出: fingerprint

1. 若文件大小 ≤ 4KB → 直接 BLAKE3 全量
2. 否则:
   a. 读取头部 4KB → SHA-1 → 写入 quick_hash 索引
   b. 命中重复 → 读取尾部 4KB → SHA-1 → 写入 quick_hash_2
   c. 仍命中 → 全量 BLAKE3 → 作为最终 fingerprint
3. 写入 files 表
```

**收益**:99% 的文件只读 4KB 即可判定不重复,大幅降低 IO。

### 6.2 风险评分公式

```
risk_score = clamp(
    rule_base                        // 规则库给的基础分(0-100)
  + system_path_bonus                // 在系统目录 +30
  + recent_access_bonus              // 近 7 天访问过 +25
  + user_pinned_bonus                // 用户 pin +50
  - ai_safe_signal                   // AI 判定为缓存类 -20
  - explicit_cache_signal,           // 路径包含 'cache' / 'tmp' -15
  0, 100
)
```

最终用 `risk_score` 映射到 risk_level:
- 0–20: low
- 21–60: medium
- 61–100: high

### 6.3 LLM 批量编排

```
1. 收集所有 Unknown / Ambiguous 文件,按归属应用聚类
2. 同一应用的文件优先打包(LLM 一次理解一组上下文)
3. 每包 50 文件,并发度 4(避免触发供应商 RPM 限制)
4. 失败重试:指数退避(1s/3s/9s)
5. 全部完成 → 批量写入 ai_cache + file_entries
```

### 6.4 增量扫描

```
扫描前: SELECT path, size, mtime FROM files INTO HashMap

遍历每个文件:
  if hashmap.contains((path, size, mtime)):
      复用上次的 fingerprint 与分类
  else:
      重新哈希 + 分类 + (必要时)AI 判定
```

### 6.5 重复文件分组算法

```
SELECT fingerprint, COUNT(*) c, SUM(size) total
FROM files
GROUP BY fingerprint
HAVING c > 1
ORDER BY total DESC;
```

UI 呈现"保留 1 个最新,其余建议清理"。

## 8. 安全与隐私设计

### 8.1 删除沙箱(再强调)

> **DiskMind 绝不直接调用 `unlink`,所有删除操作的第一步都是移入应用 Trash。**

- 30 天保留期不可缩短(产品红线)
- "立即真删"按钮需要用户**二次输入"确认"二字**才执行
- 每次真删生成不可篡改的操作日志

### 8.2 白名单(硬编码 + 用户)

**硬编码不可删除清单**(部分):
- macOS: `/System`, `/Library/Apple`, `~/Library/Keychains`, `~/Library/Mobile Documents`, 任何 `.app` Bundle 自身
- Windows: `C:\Windows`, `C:\Program Files`(应用本体), 注册表 hive 文件, `pagefile.sys`, `hiberfil.sys`

### 8.3 数据上传透明审计

- Settings → 隐私 → 「上传字段」页面,列出所有发往云端 LLM 的字段
- 用户可逐项关闭(关闭 path 后,AI 判定准确率会下降并明确提示)
- 每次扫描的"AI 调用日志"完整保留,用户可查询历史外发记录

### 8.4 API Key 加密存储

- 使用平台 keychain 存储 API Key
  - macOS: Keychain Services
  - Windows: Credential Manager
- 内存中只在请求时短暂解密
- 应用退出时清空

### 8.5 权限模型(macOS)

- Full Disk Access:**仅在用户主动扫描受保护路径时引导授权**
- 不申请通讯录、相机、麦克风等无关权限

### 8.6 自动更新签名

- macOS:Apple Developer ID 签名 + Notarization
- Windows:EV Code Signing 证书签名
- 更新包通过 HTTPS 下载,本地校验 ed25519 签名后才安装
