- **目标**:`std::panic::set_hook` 把 panic 写入 `app_data_dir/crash.log`,带时间戳 + backtrace + 进程上下文
- **执行步骤**:
  1. `lib.rs::run` 顶部调用 `std::panic::set_hook(Box::new(panic_handler))`
  2. `panic_handler` 拼接 panic info + `std::backtrace::Backtrace::capture()` + 写入 `app_data_dir/crash.log`(append 模式 + 日期分割)
  3. 提供 IPC `read_crash_log` 让前端能展示
  4. 设置 → 通用 加"打开崩溃日志目录"按钮(复用 `reveal_in_explorer`)
- **风险**:`Backtrace::capture()` 在 release build 默认禁用,需要 `RUST_BACKTRACE=1` 或 `Cargo.toml` 配置 `debug = true`
- **进阶**:用户同意后,加 Sentry SDK / 自建上报 endpoint(留作 S13 单独决策)

#### S7 详解:前端 onErrorCaptured

- **目标**:`App.vue` 顶层 `onErrorCaptured((err, instance, info) => { notify.error(...); logToFile(err); return false })` 拦截所有 Vue 组件树异常
- **执行步骤**:
  1. `App.vue` 加 `onErrorCaptured` 钩子
  2. 走 `notify.error()`(已实现)显示 toast
  3. 可选:也写入 `app_data_dir/frontend-error.log`(借助 `tauri` IPC)
- **与 S6 合并**:统一一个"崩溃 / 异常落本地"接口,前后端共用

#### S14 详解:重复文件检测(BLAKE3 两阶段)

- **目标**:解锁 `ScanOptionsCard.vue` 的 `detectDuplicates` 开关(当前是装饰)
- **算法**:两阶段哈希降低 IO 成本
  1. **第一阶段**:相同 size 的文件分组(scan 时已记录 `size_bytes`)
  2. **第二阶段(可选)**:对 size 相同的候选,读前 64KB(BLAKE3 head hash)再分组
  3. **第三阶段**:对剩余仍冲突的,读全文件 BLAKE3 确认
- **执行步骤**:
  1. 后端:`scanner/dedup.rs` 新增模块,接 `Vec<ScanResultRow>` 返回 `Vec<DuplicateGroup>`
  2. `ScanResultRow` 加可选 `hash_head` / `hash_full` 字段
  3. DB:`scan_result` 加两列(或新建 `dup_group` 表)
  4. 前端:Scan 页 Tab 加"重复文件"视图(reka-ui Tree 按 group 展开)
  5. 开关接真:`computeHash` + `detectDuplicates` 之一开启时,scan 完毕后跑 dedup pipeline
- **风险**:大量大文件场景下 IO 成本飙升,需限速 + 进度反馈
- **依赖库**:`blake3` crate(已是事实标准,~5KB binary 增量)

#### S15 详解:备份/导出 `scan_run`

- **目标**:`reports` 页或设置页加"导出扫描历史"按钮,选择 JSON / CSV 格式
- **执行步骤**:
  1. 后端:`export_scan_runs(format: "json" | "csv") -> String` IPC,返回完整 dump
  2. 前端:tauri `save` dialog 选保存路径 + toast 反馈
  3. CSV 字段:`run_id, started_at, finished_at, duration_ms, file_count, reclaimable_bytes, fingerprint`
- **可与 S9 合并**:同样思路实现 AI 日志导出 + 扫描历史导出,共享底层 CSV 生成 helper

**Sprint 3 总工作量**:~6-9 天(看是否完整做 S14)· **目标:M5 完整收官 + M6 进度 30-50%**
