**Round 24 增量交付 (2026-05-29 · Tree 视图扁化虚拟化:展开状态外提 + 单一来源)**:Round 23 把 ScanResultsTable 切到 `@tanstack/vue-virtual` + 默认 listMode 改 'table' 后主路径秒开,但 tree 视图本身仍是隐患 — `TreeNode.vue` 第 47 行 `const open = ref(props.depth < 1 || !!props.defaultOpen)` 强制 `depth=0` 永远默认开,加上递归自渲染,用户切到 tree 视图(或 advice tier-jump 没自动落 table 时)首层一展开仍会同步 mount 整片子树几千个 reka-ui 组件实例。Round 23 把 default-open 收紧到 false 只是表面缓解,深层根治需要把 tree 视图也虚拟化。本轮按"扁化 + useVirtualizer + 受控 expand"重构:

1. **新增 `lib/flattenTree.ts` + 7 个单测** — 纯函数 `flattenTree(root, expandedIds: Set<string>): FlatTreeRow[]`,把嵌套 TreeNode 按 expandedIds 展开成线性 DFS 数组(每行含 `id / node / depth / hasChildren / isExpanded`)。`nodeKey` 用 `fullPath || name` 作 stable key,collapse-single-child-dirs 后 fullPath 仍唯一。7 个测试覆盖:空树 / 默认折叠只渲染顶层 / 展开顶层后 DFS 序 / 中间层折叠不显示孙节点 / 文件 hasChildren=false / depth 严格递增 / toggle 展开仅露出直接子,锁住虚拟化前置不变量

2. **`TreeNode.vue` 改造为受控单行渲染器** — 移除内部 `open: ref` + `toggle()`,改接收 `isExpanded: boolean` prop;移除模板末尾递归 `<TreeNode v-for>` 块;删 `defaultOpen` prop,新增 `toggleExpand: [nodeId: string]` emit。从"自渲染整子树"退化为"渲染单行 + 把 toggle 意图上抛"

3. **`ScanResultsTree.vue` 切到 `useVirtualizer`** — 顶层持有 `expandedIds: ref(new Set<string>())` 单一来源;`watch(tree, immediate)` 在数据重算时把 root.children(depth=0)整批默认展开,保持原"进 tree 看到一级目录"体感;`flatNodes = computed(() => flattenTree(tree.value, expandedIds.value))`;`useVirtualizer({ count: flatNodes.length, estimateSize: 40, overscan: 8 })`,容器 `h-[calc(100vh-280px)] min-h-[420px]` 与 ScanResultsTable 对齐。`onToggleExpand(id)` 整体替换 Set(`new Set(prev)` + add/delete),触发 computed 重算 + virtualizer 自动刷新。**深层 children 不再递归实例化**,5000 节点 tree 也只创建可见 ~20 行 TreeNode,与 table 视图同档表现

4. **scan/index.vue 默认 listMode 仍保持 'table'** — table 是文件管理器主流默认 + 排序更易用 + Round 23 后用户已习惯,tree 切换为可选高级视图。但虚拟化后切到 tree 也秒开,advice tier-jump 自动切 'table' 的逻辑保留(用户在 tree 时点 tier 卡片仍会落到 virtualized table,无歧义)

**Round 8 — Trash 30 天自动清理 + 边界澄清** (2026-05-26)
- **用户疑问 → 设计澄清**:确认 `scan_result` 与 `trash_item` 是**两套独立生命周期**(后者无外键、与 scan_run 无 CASCADE 关系),滚动保留扫描历史不会影响沙箱中的回收站记录,这是有意为之 — 扫描数据是事件流可丢、回收里的文件是用户委托保管的资产不能丢
- **`30 × 500` 含义澄清**:500 是 classifier 的 `rows.truncate(500)` 风险候选上限(只命中规则 + 大小 ≥ 1MB 才入选,按物理大小降序取前 500),不是用户磁盘总文件数。典型家目录每次入库 50~200 行,30 次 ~ 几千行
- **30 天自动清理实现**(填上 1.6 唯一空白):
 - 常量:`TRASH_RETENTION_DAYS=30` / `TRASH_CLEANUP_INTERVAL_SECS=3600`
 - `db::trash_list_stale(cutoff_ms)` 查 `status='in_trash' AND moved_at < cutoff`

 - 日志:仅 purged > 0 时 `log::info!` 输出"purged N items"

**Round 7 — 反复扫描的 DB 膨胀治理 + 诊断 IPC** (2026-05-26)
- **问题发现**:用户发现反复扫描同一目录会让 `scan_run` 每次 +1、`scan_result` 每次 +500 行(`save_scan` 是纯追加,无去重无截断),DB 会无限增长
- **方案选择**:对比了「YAGNI 不动 / 滑动保留 N 次 / 内容指纹去重」三个方案,选 **B 滑动保留**(简单、立即生效、保留趋势数据语义)
- **实现**:在 `save_scan` 事务提交前加 1 行 `DELETE FROM scan_run WHERE id NOT IN (SELECT id FROM scan_run ORDER BY id DESC LIMIT MAX_SCAN_HISTORY)`,`MAX_SCAN_HISTORY=30`(常量),依赖现有 `ON DELETE CASCADE` 自动连带清 `scan_result / dir_summary`
- **诊断 IPC**:新增 `db_stats` 命令 + `DbStats` 类型(6 表行数 + db 文件字节数 + 当前 max_scan_history),`db_path` 存到 `ScanState` 供查询 size
- **数学验证**:最坏 30 × (1+500+12) = 15390 行 × ~100 bytes ≈ 1.5 MB,完全可控
- **不做**:UI 设置项(留作 P1),内容指纹去重(破坏「每次扫描是独立事件」语义,过度工程)
