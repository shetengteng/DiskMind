## 1. 一句话定义

**DuckDB 是"用 SQL 写分析查询的 SQLite"**。

- 像 SQLite 一样:**嵌入式**(单文件 / 进程内 / 零运维 / 零配置)
- 像 ClickHouse / DuckDB 一样:**列式存储 + 向量化执行**,适合 OLAP 分析查询
- 像 Pandas 一样:**直接查 CSV / Parquet / JSON**,无需先 ETL

官网:[https://duckdb.org](https://duckdb.org)
仓库:[https://github.com/duckdb/duckdb](https://github.com/duckdb/duckdb)
许可证:MIT
诞生于:荷兰 CWI 研究所(数据库领域的"贝尔实验室")

## 2. 核心特性

### 2.1 嵌入式 · 零运维
和 SQLite 一样,**单个 ~30MB 动态库**直接链入应用进程。没有 server、没有端口、没有配置文件。打开数据库 = 打开一个 `.duckdb` 文件,关闭 = `close()`。

### 2.2 列式存储 + 向量化执行
- 数据按**列**而非按行物理存储 → 聚合 / 分组 / 过滤大表时只读相关列
- 一次处理 1024 行的向量(SIMD 加速)→ 比传统行式 OLTP 快 **10–100×**
- 典型 1 亿行的 `SUM` / `GROUP BY` 操作秒级返回

### 2.3 完整 SQL 支持
- 标准 ANSI SQL + 窗口函数 + CTE + 递归 CTE
- 高阶函数(`LIST_TRANSFORM`、`UNNEST`)
- JSON 原生函数(`json_extract`、`json_each`)
- 时间函数丰富(`date_trunc`、`time_bucket`)

### 2.4 多文件格式直读(杀手锏)
```sql
-- 直接查 Parquet,不需要导入
SELECT * FROM 'logs/*.parquet' WHERE level = 'ERROR';

-- 直接查 CSV
SELECT count(*) FROM read_csv_auto('orders.csv');

-- 直接查远程 URL
SELECT * FROM 'https://example.com/data.parquet';
```
没有 schema 定义、没有 LOAD DATA、没有 import 步骤。

### 2.5 与 SQLite / Postgres 互操作
```sql
-- 直接读 SQLite
INSTALL sqlite; LOAD sqlite;
SELECT * FROM sqlite_scan('./diskmind.db', 'files');

-- 直接读 Postgres
INSTALL postgres; LOAD postgres;
SELECT * FROM postgres_scan('host=... dbname=...', 'public', 'orders');
```

### 2.6 客户端语言绑定
官方支持:**Python · R · Rust · Node.js · Java · Go · C/C++ · Swift · WASM**
其中 WASM 版本可在浏览器里跑完整 OLAP。

## 3. 与同类技术对比

| 维度 | SQLite | DuckDB | Postgres | ClickHouse |
| --- | --- | --- | --- | --- |
| 部署形态 | 嵌入式 | **嵌入式** | C/S | C/S |
| 存储模型 | 行式 | **列式** | 行式 | 列式 |
| 适用场景 | OLTP / 小数据 | **OLAP / 嵌入式分析** | OLTP / 通用 | 大规模 OLAP |
| 单文件 | ✓ | ✓ | ✗ | ✗ |
| 并发写 | 单写者 | 单写者 | 强 | 强 |
| 分析查询性能 | 慢 | **极快** | 中 | 极快 |
| 安装大小 | < 5MB | ~30MB | > 100MB | > 200MB |
| 直读 Parquet/CSV | ✗ | **✓** | 需扩展 | ✓ |
| 学习曲线 | 低 | 低 | 中 | 中-高 |

**一句话总结**:DuckDB = SQLite 的体验 + 列式 OLAP 的性能。

## 4. 为什么 DiskMind 选 DuckDB

DiskMind 的数据有两种**完全不同的访问模式**,需要两类数据库分工:

| 访问模式 | 例子 | 选型 | 理由 |
| --- | --- | --- | --- |
| **事务型(OLTP)** | 写入文件元数据 / 标记 trash item / 更新 ai_cache | **SQLite** | 强事务、行式、单条 update 快 |
| **分析型(OLAP)** | 跨 scan 趋势统计 / Top 应用 / 重复文件聚合 / 报告页所有图表 | **DuckDB** | 大表聚合秒级、内存友好、SQL 表达力强 |

### 4.1 不选 SQLite 跑分析查询的理由
DiskMind 单次扫描可能产出 **数百万行 file_entries**,跨 10 次扫描就有上千万行。
- SQLite 跑 `SUM(size) GROUP BY day FROM 10M+ rows` 需要 **5–30 秒**
- DuckDB 同样查询通常 **< 1 秒**
- 用户在"报告页"切换时间范围时不能等

### 4.2 不选 ClickHouse / Postgres 的理由
- 都是 C/S 架构,需要单独进程 → 与"零运维桌面应用"哲学冲突
- 安装包额外 100MB+ → 影响 Tauri 总包体积
- 跨平台二进制分发复杂(尤其 Windows)

### 4.3 不选纯 Pandas / Polars 的理由
- Rust 生态里 Polars 优秀,但 **DuckDB 直接读 SQLite 文件**这个能力杀手级
- SQL 比 DataFrame API 更**用户友好 / 可调试**
- DuckDB 内置磁盘溢出,Polars 默认全内存,大表会 OOM

## 5. 在 DiskMind 中的具体应用

### 5.1 分析视图(Materialized View)

```sql
-- 跨 scan 释放趋势(报告页折线图)
CREATE VIEW v_scan_trends AS
SELECT
    date_trunc('day', to_timestamp(started_at)) AS day,
    SUM(scanned_bytes) / 1024 / 1024 / 1024 AS scanned_gb,
    COUNT(*) AS scan_count
FROM scans
WHERE status = 'completed'
GROUP BY day;

-- Top 占用应用(报告页柱状图)
CREATE VIEW v_top_apps AS
SELECT
    f.owner_app,
    SUM(f.size) AS total_bytes,
    COUNT(*) AS file_count
FROM files f
WHERE owner_app IS NOT NULL
GROUP BY owner_app
ORDER BY total_bytes DESC
LIMIT 50;

-- 重复文件聚合(扫描结果"重复"Tab)
CREATE VIEW v_duplicates AS
SELECT
    fingerprint,
    COUNT(*) AS copies,
    SUM(size) AS total_bytes,
    LIST(path ORDER BY mtime DESC) AS paths
FROM files
GROUP BY fingerprint
HAVING copies > 1
ORDER BY total_bytes DESC;
```

### 5.2 同时读 SQLite + 自有列式存储(混合模式)

```sql
-- 把 SQLite 中的事务数据 attach 进来,直接用 DuckDB 跑分析
INSTALL sqlite; LOAD sqlite;
ATTACH 'diskmind.db' AS oltp (TYPE sqlite);

-- 一条查询同时跨两个引擎
SELECT
    s.id,
    s.started_at,
    COUNT(fe.fingerprint) AS file_count,
    SUM(f.size) AS total_size
FROM oltp.scans s
JOIN oltp.file_entries fe ON fe.scan_id = s.id
JOIN oltp.files f ON f.fingerprint = fe.fingerprint
WHERE s.started_at > epoch(NOW() - INTERVAL 30 DAY)
GROUP BY s.id, s.started_at
ORDER BY s.started_at DESC;
```

> 这一招的实战价值:**不需要为分析复制一份数据**,SQLite 是 OLTP 的唯一真相来源,DuckDB 只是"分析视图层"。

### 5.3 物化重型查询(可选优化)

如果某个分析视图被频繁查询,可以物化:

```sql
-- 创建物化表
CREATE TABLE mv_app_trends AS
SELECT
    f.owner_app,
    date_trunc('day', to_timestamp(s.started_at)) AS day,
    SUM(f.size) AS bytes
FROM oltp.scans s
JOIN oltp.file_entries fe ON fe.scan_id = s.id
JOIN oltp.files f ON f.fingerprint = fe.fingerprint
GROUP BY f.owner_app, day;

-- 后台定期刷新(简单做法:DROP + CREATE)
```

## 6. Rust 集成示例

DiskMind 使用 `duckdb` crate(官方维护)。

### 6.1 添加依赖
```toml
# Cargo.toml
[dependencies]
duckdb = { version = "1.1", features = ["bundled"] }
```

### 6.2 基础用法

```rust
use duckdb::{Connection, Result, params};

fn open_analytics(path: &str) -> Result<Connection> {
    let conn = Connection::open(path)?;
    conn.execute_batch(r"
        INSTALL sqlite;
        LOAD sqlite;
        ATTACH 'diskmind.db' AS oltp (TYPE sqlite);
    ")?;
    Ok(conn)
}

fn top_apps(conn: &Connection, limit: usize) -> Result<Vec<(String, u64)>> {
    let mut stmt = conn.prepare(r"
        SELECT owner_app, SUM(size) AS total
        FROM oltp.files
        WHERE owner_app IS NOT NULL
        GROUP BY owner_app
        ORDER BY total DESC
        LIMIT ?1
    ")?;
    let rows = stmt.query_map(params![limit], |row| {
        Ok((row.get::<_, String>(0)?, row.get::<_, u64>(1)?))
    })?;
    rows.collect()
}
```

### 6.3 流式读大结果集

```rust
let mut stmt = conn.prepare("SELECT * FROM v_duplicates")?;
let mut rows = stmt.query([])?;
while let Some(row) = rows.next()? {
    // 处理每一行,不会一次性 load 进内存
}
```

## 7. 注意事项与限制

| 注意点 | 说明 |
| --- | --- |
| **单写者** | DuckDB 同 SQLite,同一时刻只能一个写连接,但可有多读连接 |
| **不适合 OLTP 高并发写** | 设计目标是分析,不是事务交易系统 |
| **类型严格** | SQL 类型系统比 SQLite 更严格(SQLite 是动态类型) |
| **磁盘文件锁** | 同一个 `.duckdb` 文件不能被多个进程同时写 |
| **WAL** | 1.0+ 已支持 WAL,但与 SQLite 实现不同 |
| **崩溃恢复** | 1.x 已稳定,但建议关键事务仍走 SQLite |
| **包体积** | bundled 模式约 +30MB,可考虑动态加载 |

## 9. 在 DiskMind 项目中的最终定位

> **一句话**:SQLite 是 DiskMind 的"账本",DuckDB 是 DiskMind 的"数据分析师"。

| 引擎 | 在 DiskMind 中的角色 |
| --- | --- |
| **SQLite** | 主存储 · 写入文件元数据 / scans / trash_items / ai_cache / 配置 · 事务性强、写入频繁 |
| **DuckDB** | 只读分析层 · 报告页所有图表 / 重复文件聚合 / 趋势分析 · 直接 ATTACH SQLite,不复制数据 |
| **协作** | DuckDB 全程"借"SQLite 的数据做分析,SQLite 保持单一真相来源(SSOT) |

这种分工的本质是 **CQRS(Command Query Responsibility Segregation)** 在嵌入式场景下的最小成本实现:写走 SQLite,分析读走 DuckDB,无需任何同步管道。
