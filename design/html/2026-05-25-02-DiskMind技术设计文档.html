3.1 Scanner — 扫描引擎

3.1.1 职责

  • 遍历指定目录,产出文件元数据流
  • 计算文件指纹(分级哈希)
  • 写入本地元数据索引
  • 支持取消、暂停、增量

3.1.2 调度策略

根据存储介质区分(关键设计):

  • SSD:并行度 = num_cpus,最大化吞吐
  • HDD:并行度 = 1–2,避免随机读放大延迟
// 伪代码
let strategy = match probe_disk_type(&root) {
    DiskType::Ssd => ScanStrategy { parallelism: num_cpus(), chunk_size: 4096 },
    DiskType::Hdd => ScanStrategy { parallelism: 2,           chunk_size: 1024 },
};

3.1.3 遍历策略

  • BFS 优先,内存中维护队列上限(防爆栈)
  • 跳过清单(硬编码 + 用户白名单):
    • macOS: /System, /private, /Volumes
    • Windows: C:\Windows\System32, C:\$Recycle.Bin, pagefile.sys
  • 软链接处理:默认不跟随,避免循环

3.1.4 哈希策略(分级)

文件 ≤ 4 KB → 全量 BLAKE3 (快) 4 KB < ≤ 64 MB → 头部 4KB SHA-1 + 尾部 4KB SHA-1 → 命中再算 BLAKE3 > 64 MB → 头/中/尾各 4KB SHA-1 + size → 命中再 BLAKE3

理由:绝大多数重复文件大小不同,二阶哈希可避免不必要的全量计算。

3.1.5 增量扫描

  • 上次扫描的 (mtime, size) 作为 cache key
  • 未变 → 复用上次的指纹与分类结果
  • 变化 → 重新计算

3.1.7 输出数据结构

pub struct FileEntry {
    pub path: PathBuf,
    pub size: u64,
    pub mtime: i64,
    pub atime: i64,
    pub fingerprint: Option<String>,   // BLAKE3 hex
    pub mime_guess: Option<String>,
    pub owner_app: Option<String>,     // 通过路径推断的归属应用
    pub category: Option<Category>,    // 由 Classifier 填充
}

3.3 AIEngine — AI 引擎

3.3.1 双轨切换

enum AiBackend {
    Cloud(CloudConfig),   // DeepSeek / GPT / Claude
    Local(OllamaConfig),  // ollama HTTP API
    Hybrid,               // 元数据本地预筛 → 复杂决策上云
}

用户在设置中切换,默认 Hybrid。

3.3.2 批量打包策略

为降低 token 成本,单次请求评估 50 个文件:

请求体 ≈ system_prompt (固定 800 tokens) + 50 × 80 tokens = ~4800 tokens 响应体 ≈ 50 × 120 tokens = ~6000 tokens 单次成本 ≈ ¥0.03 (DeepSeek-V3)

3.3.3 Prompt 模板(System,节选)

你是 DiskMind 的文件分析助手。基于以下文件元数据,判断每个文件的类别、
删除风险与用户面向解释。严格输出 JSON,符合 schema。

【输入】文件数组:[{path, size_mb, mtime, owner_app_guess, mime}]

【输出 Schema】:
[{
  "path": string,
  "category": "AppCache" | "AppLog" | "TempFile" | "MediaCache"
            | "DeveloperJunk" | "DownloadedInstaller" | "Unknown",
  "risk_score": 0-100,
  "risk_level": "low" | "medium" | "high",
  "explanation": string (≤ 60 字中文),
  "action": "safe_to_delete" | "review" | "keep" | "unknown"
}]

【规则】
1. 证据不足返回 category="Unknown" + action="unknown",不要臆造
2. 系统目录、用户文档目录 → risk_score >= 80
3. 解释要让普通用户看得懂,避免技术黑话
4. 严禁建议删除任何与近 7 天访问相关的文件

3.3.4 JSON Schema 校验

使用 schemars + jsonschema crate 做硬约束。校验失败 → 重试一次 → 仍失败 → 降级为规则模式默认值。

3.3.5 降级链

Cloud LLM 失败 → 自动重试 1 次 → 切换备选 LLM → 降级为本地 Ollama → 仍失败 → 仅返回规则库结果,UI 提示"AI 暂不可用"

3.4 Policy — 决策引擎

3.4.1 三档策略

策略默认勾选适用
Safe(安全)risk_score ≤ 20默认推荐给新手
Balanced(谨慎)risk_score ≤ 50推荐给中级用户
Aggressive(激进)risk_score ≤ 80 但逐项确认极客使用

3.4.2 白名单(永不删除)

  • 系统受保护目录
  • 用户近 7 天打开过的文件
  • 用户手动 pin 的文件
  • 标记为"重要"的文件
  • 路径包含 key / license / token 等关键字的小文件

3.5 Cleaner — 清理执行器

⚠ 产品红线

DiskMind 自己不直接调用 unlink/DeleteFile,而是先移入应用内 Trash。

3.5.1 沙箱机制

用户点击"清理" → Cleaner 在 ~/.diskmind/trash/<bundle_id>/ 下创建硬链接(或移动) → 在 SQLite 中记录 trash_manifest: { original_path, trash_path, deleted_at, scan_bundle_id } → 显示"已清理 X.XX GB,可在 Trash 中恢复" → 30 天后,后台 worker 异步真删

3.5.2 回滚

  • UI 提供"Trash"视图,列出 30 天内所有"已清理"项
  • 单项恢复:rename 回原路径,若原路径已被占用 → 提示用户重命名
  • 批量恢复:按 scan_bundle_id 一次回滚整个扫描会话

3.5.4 异常处理

异常行为
文件被占用标记 pending,下次重试
权限不足UI 提示需提权,引导用户授权
路径不存在视为已清理,从 manifest 移除
跨卷失败改用复制+删除策略

4.1 ER 概览

+---------+ +---------------+ +----------+ | scans |1---*| file_entries |*---1 | files | +---------+ +-------+-------+ +----------+ | | +----+----+ +-----------+ |ai_cache | |trash_items| +---------+ +-----------+

4.2 SQLite DDL(核心表)

-- 扫描会话
CREATE TABLE scans (
  id            TEXT PRIMARY KEY,
  started_at    INTEGER NOT NULL,
  finished_at   INTEGER,
  root_paths    TEXT NOT NULL,        -- JSON
  status        TEXT NOT NULL,        -- running/completed/cancelled/failed
  total_files   INTEGER,
  scanned_bytes INTEGER
);

-- 文件元数据(主表,跨 scan 共享)
CREATE TABLE files (
  fingerprint   TEXT PRIMARY KEY,
  path          TEXT UNIQUE NOT NULL,
  size          INTEGER NOT NULL,
  mtime         INTEGER NOT NULL,
  atime         INTEGER,
  mime_guess    TEXT,
  owner_app     TEXT
);
CREATE INDEX idx_files_path ON files(path);
CREATE INDEX idx_files_size ON files(size);

-- 扫描中的文件实例
CREATE TABLE file_entries (
  scan_id       TEXT NOT NULL,
  fingerprint   TEXT NOT NULL,
  category      TEXT NOT NULL,
  risk_score    INTEGER,
  risk_level    TEXT,
  explanation   TEXT,
  action        TEXT,
  source        TEXT NOT NULL,        -- rule/ai/cache
  PRIMARY KEY (scan_id, fingerprint),
  FOREIGN KEY (scan_id) REFERENCES scans(id),
  FOREIGN KEY (fingerprint) REFERENCES files(fingerprint)
);

-- AI 结果缓存
CREATE TABLE ai_cache (
  cache_key     TEXT PRIMARY KEY,     -- sha1(fingerprint + model + prompt_hash)
  result_json   TEXT NOT NULL,
  model         TEXT NOT NULL,
  created_at    INTEGER NOT NULL
);

-- Trash 沙箱
CREATE TABLE trash_items (
  id            TEXT PRIMARY KEY,
  scan_id       TEXT,
  original_path TEXT NOT NULL,
  trash_path    TEXT NOT NULL,
  size          INTEGER NOT NULL,
  deleted_at    INTEGER NOT NULL,
  hard_delete_after INTEGER NOT NULL,
  status        TEXT NOT NULL          -- in_trash / hard_deleted / restored / pending
);

4.3 DuckDB 分析视图

-- 跨 scan 趋势
CREATE VIEW v_scan_trends AS
SELECT
    date_trunc('day', to_timestamp(started_at)) AS day,
    SUM(scanned_bytes) / 1024 / 1024 / 1024 AS scanned_gb,
    COUNT(*) AS scan_count
FROM scans
WHERE status = 'completed'
GROUP BY day;

-- Top 占用应用
CREATE VIEW v_top_apps AS
SELECT
    f.owner_app,
    SUM(f.size) AS total_bytes,
    COUNT(*) AS file_count
FROM files f
WHERE owner_app IS NOT NULL
GROUP BY owner_app
ORDER BY total_bytes DESC
LIMIT 50;

06关键算法

去重、风险评分、批量编排、增量扫描 — 性能与准确性的核心。

6.1 分级哈希(去重)

1. 若文件大小 ≤ 4KB → 直接 BLAKE3 全量 2. 否则: a. 读取头部 4KB → SHA-1 → 写入 quick_hash 索引 b. 命中重复 → 读取尾部 4KB → SHA-1 → 写入 quick_hash_2 c. 仍命中 → 全量 BLAKE3 → 作为最终 fingerprint 3. 写入 files 表
收益

99% 的文件只读 4KB 即可判定不重复,大幅降低 IO。

6.2 风险评分公式

risk_score = clamp(
    rule_base                        // 规则库给的基础分(0-100)
  + system_path_bonus                // 在系统目录 +30
  + recent_access_bonus              // 近 7 天访问过 +25
  + user_pinned_bonus                // 用户 pin +50
  - ai_safe_signal                   // AI 判定为缓存类 -20
  - explicit_cache_signal,           // 路径包含 'cache' / 'tmp' -15
  0, 100
)

最终映射:

  • 0–20 low · 默认勾选
  • 21–60 medium · 需用户确认
  • 61–100 high · 默认不动

6.3 LLM 批量编排

1. 收集所有 Unknown / Ambiguous 文件,按归属应用聚类 2. 同一应用的文件优先打包(LLM 一次理解一组上下文) 3. 每包 50 文件,并发度 4(避免触发供应商 RPM 限制) 4. 失败重试:指数退避(1s/3s/9s) 5. 全部完成 → 批量写入 ai_cache + file_entries

6.4 增量扫描

扫描前: SELECT path, size, mtime FROM files INTO HashMap

遍历每个文件:
  if hashmap.contains((path, size, mtime)):
      复用上次的 fingerprint 与分类
  else:
      重新哈希 + 分类 + (必要时)AI 判定

6.5 重复文件分组

SELECT fingerprint, COUNT(*) c, SUM(size) total
FROM files
GROUP BY fingerprint
HAVING c > 1
ORDER BY total DESC;

UI 呈现"保留 1 个最新,其余建议清理"。

08安全与隐私设计

五条不可触碰的产品红线,加上完整的隐私透明度。

⚠ 五大安全红线
  1. DiskMind 绝不直接 unlink,所有删除先入应用 Trash
  2. 30 天保留期不可缩短(产品红线)
  3. 文件内容绝不上传 — 仅元数据(路径/大小/时间/MIME)
  4. 删除路径代码必须 2 人评审
  5. Trash retention 修改需 CTO 审批

8.1 白名单(硬编码 + 用户)

硬编码不可删除清单(部分):

  • macOS:/System, /Library/Apple, ~/Library/Keychains, ~/Library/Mobile Documents,任何 .app Bundle 自身
  • Windows:C:\Windows, C:\Program Files(应用本体),注册表 hive 文件, pagefile.sys, hiberfil.sys

8.2 数据上传透明审计

  • Settings → 隐私 → 「上传字段」页面,列出所有发往云端 LLM 的字段
  • 用户可逐项关闭(关闭后 AI 准确率会下降,明确提示)
  • 每次扫描的"AI 调用日志"完整保留,可查询历史外发记录

8.3 API Key 加密存储

  • 使用平台 keychain:
    • macOS: Keychain Services
    • Windows: Credential Manager
  • 内存中只在请求时短暂解密
  • 应用退出时清空

8.4 权限模型(macOS)

  • Full Disk Access:仅在用户主动扫描受保护路径时引导授权
  • 不申请通讯录、相机、麦克风等无关权限

8.5 自动更新签名

  • macOS:Apple Developer ID 签名 + Notarization
  • Windows:EV Code Signing 证书签名
  • 更新包通过 HTTPS 下载,本地校验 ed25519 签名后才安装
