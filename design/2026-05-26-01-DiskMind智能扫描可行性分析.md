# DiskMind 智能扫描可行性分析

> 文档版本:v1.0 · 撰写日期:2026-05-26 · 状态:Draft · 关联:[2026-05-25-02-技术设计文档](./2026-05-25-02-DiskMind技术设计文档.md) / [2026-05-25-05-开发待办](./2026-05-25-05-DiskMind开发待办.md)

---

## 0. 一句话结论

**可行,且实现成本低**(MVP 2–3 天)。本质是给已有 `scan_result` 表加一个「自然语言 → DSL → SQL」前置层,不动 Scanner / DB schema,复用现有结果渲染管线。**最大不确定性不在技术,在意图歧义**(用户说"最近"到底是 7 天还是 30 天)。

---

## 1. 需求一句话

用户在 AI Drawer / 全局搜索里说人话:

> "帮我找一下最近的视频"
> "已经一年没动过的图片"
> "Downloads 里大于 1G 的文件"
> "上周下载的安装包"

系统直接返回精准结果,**不需要用户手动配过滤器**。

---

## 2. 核心矛盾(第一性原理)

| 现状 | 目标 |
|---|---|
| 用户先勾扫描目录 → 跑全量扫描 → 在结果页用 search / 风险 / 分类 3 个过滤器手动筛 | 用户说一句话,系统自动定位目录 + 自动加过滤器 + 直出结果 |
| 过滤维度只有 3 个(关键字 / 风险 / 分类) | 至少要支持 **时间、类型、大小、目录** 4 维任意组合 |
| Scanner 是「一次全量遍历」模型,结果落 SQLite | 智能扫描不应每次都跑 IO,而是**在已扫描结果上做 query**(全量扫描每周/手动触发一次即可) |

**关键洞察:**「智能扫描」**不是**扫描器升级,**是**给 `scan_result` 表加一个 NL Query 层。

## 4. Query DSL Schema(MVP 完整版)

```json
{
  "intent": "find_files | summarize | suggest_cleanup",
  "filters": {
    "extensions": ["mp4", "..."],
    "categories": ["视频", "..."],
    "risk_in": ["high", "medium"],
    "mtime_after": "2026-05-19T00:00:00Z",
    "mtime_before": null,
    "atime_after": null,
    "size_min_bytes": 1073741824,
    "size_max_bytes": null,
    "path_contains": "Downloads",
    "path_starts_with": null,
    "path_excludes": [".git", "node_modules"]
  },
  "sort": {
    "by": "mtime | size | path",
    "order": "asc | desc"
  },
  "limit": 50,
  "explanation": "string,用户能看懂的中文说明"
}
```

**4 维任意组合**(时间 / 类型 / 大小 / 目录),涵盖 95% 自然语言场景。

## 5. 与现有架构集成点

| 模块 | 现状 | 智能扫描如何复用 / 改动 |
|---|---|---|
| **Scanner** | walkdir 全量遍历,产出 `FileEntry` | **零改动**(智能扫描只查 `scan_result`,不触发新扫描) |
| **DB scan_result** | 已存 `path/size/mtime/extension/category/risk` | **零改动**,SQL where 子句直接命中 |
| **AI Orchestrator** | 已有 chat / explain_file / cleaning_advice / batch_classify | **加 1 个场景 `smart_query`**,复用 provider/fallback/cost 记账 |
| **AI 前端 store** | `useAiStore` 已管 chat 流 + explainFile + adviceResult | **加 `smartQuery(text)` action**,产物挂到 `smartQueryResult` 状态 |
| **AI Drawer UI** | 显示 chat + 引用文件 | **加 1 个 query 结果卡片**(顶部显示 explanation + 「在结果页查看 N 个文件」CTA) |
| **Scan 页 / 结果 Table** | 接 store.results | **零改动**(智能扫描 query 命中时,把结果灌入同一 store 即可) |

## 6. AI 翻译层 — 关键 Prompt 设计

```text
你是 DiskMind 的智能扫描查询解析器。用户的自然语言意图,你必须翻译为单个 JSON 对象(Query DSL),不要 markdown 代码块。

Schema (省略,见 §4)

歧义处理规则:
- "最近" 默认为 7 天;"上周" = 7 天;"上个月" = 30 天;"很久" = 180 天
- "大文件" 默认 size_min_bytes = 100 MB
- "视频" → ext in [mp4, mov, mkv, avi, webm, flv, m4v]
- "图片" → ext in [jpg, jpeg, png, gif, webp, heic, raw, tiff]
- "安装包" → ext in [dmg, pkg, exe, msi, deb, rpm, apk]
- "缓存" → category = "缓存" OR path_contains in ["Cache", "Caches"]
- 路径用户给的不带斜杠时,放进 path_contains
- 如果意图模糊到无法翻译,intent = "find_files",filters 留空,explanation 写明
- explanation 字段必须用中文,15-40 字之间,告诉用户你理解成了什么

当前日期(用户时区):{TODAY_ISO}
```

**关键设计:** prompt 里**写死了「最近」「大文件」「视频」等模糊词的标准化映射**,把不确定性收敛到产品策略层,而不是放任 LLM 自由发挥。

## 7. 三个典型场景演练

| 用户说 | AI 翻译产物 | SQL 实际查询 |
|---|---|---|
| "帮我找一下最近的视频" | `{filters:{extensions:[mp4,mov,mkv,...], mtime_after:"7d前"}, sort:mtime desc, limit:50, explanation:"最近 7 天的视频"}` | `WHERE extension IN (...) AND mtime > ... ORDER BY mtime DESC LIMIT 50` |
| "Downloads 里大于 1G 的文件" | `{filters:{path_contains:"Downloads", size_min_bytes:1073741824}, sort:size desc, limit:50, explanation:"Downloads 里超过 1GB 的文件"}` | `WHERE path LIKE '%Downloads%' AND size > ... ORDER BY size DESC` |
| "已经一年没动过的图片" | `{filters:{extensions:[jpg,png,...], mtime_before:"365d前"}, sort:mtime asc, limit:100, explanation:"一年以上未修改的图片"}` | `WHERE extension IN (...) AND mtime < ... ORDER BY mtime ASC` |

## 8. 风险与对应策略

| 风险 | 影响 | 缓解 |
|---|---|---|
| **AI 翻译错** ("最近" 被理解为 30 天而非 7 天) | 用户拿到的结果不准 | (1) `explanation` 字段强制要求 + UI 高亮显示,让用户能立刻发现 (2) 提供「调整范围」chip 让用户一键改时间窗 (3) prompt 里固定标准映射 |
| **AI 翻译产物 JSON 不合规** | 解析失败 | Rust 侧用 `serde + JSON Schema 校验`,失败时降级到「关键词全字段 LIKE 搜索」 |
| **AI 调用慢/失败** | 用户白等 | 复用现有 fallback chain(主 provider → backup → local rules)。**本地兜底规则**:简单 grep 关键词到 ext / category / 时间词,无 AI 也能 work |
| **没有可用 scan_result** | 用户从未扫描过 | UI 引导先跑一次全量扫描 |
| **结果集太大撑爆 drawer** | UI 卡 | DSL 强制 `limit` 默认 50,上限 500;前端虚拟滚动复用现有 Table |
| **跨设备/跨 root 重复结果** | scan_result 可能有多次 run | 默认查最新一次 `run_id`,DSL 不暴露 run_id 选择(避免认知负担) |

## 10. 推荐 V2 演进路径(MVP 之后)

| 阶段 | 增量能力 |
|---|---|
| V1(本文 MVP) | NL → DSL → SQL,4 维过滤组合 |
| V2 | **语义搜索**:对文件名 + 路径建 embedding 索引,支持 "找我那个关于秋招的 pdf" 这种模糊语义 |
| V3 | **多轮对话改写**:支持 "再加上排除 git 仓库"、"按大小排" 这种增量修正 |
| V4 | **执行确认 + 安全**:对返回结果支持 "把这些移到回收站",AI 给出风险提示后用户一键 confirm |

## 11. 决策项(本文档不替你决定)

1. **入口形态**:命令栏(Cmd+K)?AI Drawer 输入框?Scan 页顶部搜索框?三选一
2. **是否要求先有 scan_result**:还是支持「现场触发轻量扫描指定目录」?
3. **本地兜底**:是否真做?LLM 可用率高的话可以砍掉,代价是离线/key 没钱时无响应

> 建议默认值:AI Drawer 入口 + 必须先有 scan_result + 做本地兜底(成本低、上限保证体验)。

---

## 附:不做什么(明确边界)

- 不做语义搜索(V2 范围)
- 不做执行类指令(删/移操作走现有 trash 流程,本期只做查询)
- 不做跨 run 的趋势分析(查最新 run_id 即可)
- 不做用户偏好学习(没必要,prompt 内固定映射更可控)
